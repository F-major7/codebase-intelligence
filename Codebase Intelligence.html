Engineering Decisions

Thoughtful tradeoffs,
not prompt engineering

Every architectural choice optimizes for cost, latency, and simplicity — with explicit non-goals to constrain scope.

01
HNSW Graph Indexing
Hierarchical Navigable Small World graphs provide O(log N) approximate nearest-neighbor search. At query time, traversal starts from sparse entry-point nodes at upper layers, greedily descending to the dense lower layer — visible in the hero animation above.
O(log N) search ChromaDB SQLite backend
02
Structure-Aware Chunking
RecursiveCharacterTextSplitter with code-specific separators prioritizes splitting at class and function boundaries before falling back to line breaks. 1000-char chunks with 200-char overlap balance precision and context continuity.
class → def → \n\n → \n 1000 chars · 200 overlap
03
Cost-Optimized Model Selection
Claude Haiku at $0.002/query vs. Sonnet at $0.024/query — a 12× cost difference with marginal quality delta for most code Q&A. Embeddings use text-embedding-3-small at $0.02/1M tokens. ChromaDB persistence eliminates re-embedding costs on repeat queries.
Haiku · $0.002 te-3-small · $0.02/1M persistent index
04
Session-Based Repo Isolation
Dynamic ChromaDB collection management enables concurrent analysis of multiple codebases without cross-contamination. Custom repos are session-scoped to manage the 1GB Streamlit Cloud storage limit — pre-loaded Flask, FastAPI, and Django collections persist permanently for instant testing.
session isolation dynamic collections Flask · FastAPI · Django
Implementation

Real code, real tradeoffs

src/ingestion/chunker.py
class CodeChunker:
  """Structure-aware code splitting."""

  def __init__(self, chunk_size=1000,
               chunk_overlap=200):
    # Code-aware separators, priority order:
    # class → function → async fn → blank line
    self.text_splitter = RecursiveCharacterTextSplitter(
      chunk_size=chunk_size,
      chunk_overlap=chunk_overlap,
      separators=[
        "\n\nclass ",    # class boundaries first
        "\n\ndef ",      # then functions
        "\n\nasync def ", # async functions
        "\n\n",          # paragraph breaks
        "\n",            # line breaks
        ""               # char-level (last resort)
      ]
    )
Why hierarchical separators?
Code has natural semantic boundaries that prose doesn't. Splitting mid-function destroys context. By trying class → function boundaries first, most chunks correspond to a complete, meaningful unit of logic.
Why 200-char overlap?
Ensures function signatures and docstrings appear in both the preceding and following chunk — so retrieval never returns context-free code snippets missing their definition header.
What was considered instead?
AST-based chunking (language-specific parsing) would be more precise but requires per-language tooling. The recursive splitter achieves ~85% of the benefit at zero configuration overhead — a deliberate tradeoff for development speed.
